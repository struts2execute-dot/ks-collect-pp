## 三、反篡改机制逆向分析

### 3.1 各文件反篡改框架

| 文件 | 框架变量 | 核心方法 | 正则位置 |

### 3.2 k2/C2/j4/q2 工作原理（列号检测）

```
C2.t4(C2.K5()[N](), N, FILE_OFFSET)
  → TypeError()生成stack trace → 正则提取列号
  → 列号 - OFFSET - 2 - 调整值 = 数组索引
  → 用于q93查找表，控制opaque条件分支
```

等长替换不影响列号→不触发k2。但某些区域被标记为"受保护"，与距离无关，只能通过实测确认。

### 3.3 H4x 工作原理（5135.js代码完整性校验）

```
Y5f[3] 累加器在代码执行过程中通过多个 Y5f[3]+=2 操作累加
  → H4x.t99() 对累加值做hash（内部调用 p5[174745].d3LhUCN）
  → hash结果与预期值 Y5f[6]/Y5f[7] 比较
  → 不匹配 → n7_[544] = truthy → 惩罚激活
  → 匹配 → n7_[544] 保持false → 正常运行
```

**与k2的区别**：H4x不是检测文件修改的列号，而是检测运行时代码执行路径的累加值hash。这意味着**其他文件的patch**可以间接导致5135.js的校验失败（通过改变异步执行时序影响Y5f[3]的最终值）。

### 3.4 k2监控实测结论

监控是**区域性的、非距离相关的**。无法通过距离预测安全性，只能实测确认。

| 类型 | 表现 | 弹出框 | v18后频率 | v19 patch后频率 |
|------|------|--------|----------|----------------|
| Type 1 | spin按钮直接卡死 | 无 | 偶发（降低） | 进一步降低 |
| Type 2 | PGSoft验证弹框+spin卡死 | 有 | 偶发（降低） | 进一步降低 |

### 5.2 v19 patch确认的卡死根因

v19通过逆向5135.js确认：偶发卡死的根本原因是**5135.js中的代码完整性校验系统**（n7_[544]惩罚标记）。其他JS文件的patch改变了异步执行时序，导致Y5f[3]累加器的hash值偶发地不匹配预期值，从而在responseCb中吞掉回调，tryAutoSpin永远不被调用。

### 5.3 v19 patch后Console日志特征

**正常状态**：
```
[Hook] BVSpinButtonController found, yb=3
[Hook] installed OK (v2 - tracks both activate and suspend)
（无ALERT，yb正常从3变为1）
```

**卡死状态（v19 patch后仍偶发）**：
```
[Hook] BVSpinButtonController found, yb=3
[Hook] installed OK (v2 - tracks both activate and suspend)
[ALERT] 5秒后yb仍为3(SUSPENDED)！激活失败！
[ALERT] 当前状态：Cb=0 yb=3
```

### 5.4 v20后仍偶发的可能原因

5个有惩罚机制的JS全部修补完成后，残余卡死可能来自：

1. **其他6个辅助JS中存在未发现的惩罚机制** — 目前仅做了死循环patch，未做深度扫描
2. **94/assets/目录下的游戏引擎JS** — 可能有独立的安全检查
3. **运行时事件系统中未发现的惩罚回调** — 通过996b的事件路由触发
4. **5135.js中可能还有其他惩罚路径** — responseCb之外的其他回调拦截点

### 5.5 v19 patch备选方案（如需进一步加固）

**方案2: 在responseCb中短路检查**

在offset 283918处，将`n7_[544]`检查改为恒假：
```
原文: if(n7_[544] && H4x.k09()...
改为: if(!1 && H4x.k09()...
```
注意：`n7_[544]` 是8字节，`!1      ` 需空格填充到8字节保持等长。

**方案3: Hook方式（不修改文件）**

在index.html中注入脚本，定时将n7_[544]重置为false。但需找到n7_数组的运行时引用，实现较复杂。
